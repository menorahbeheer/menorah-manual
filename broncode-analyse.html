2. Voorverwarming en I-tijd

Hoe het werkt

FbScheduler berekent per klok de waarde iTimeBeforeOperation (INT, minuten): positief = resterende tijd tot de volgende gebruiksperiode, negatief = resterende duur van de actieve periode. Deze waarde gaat naar FbStartStopHeatingCircuit, die bepaalt of vroegstart (xOptimization) geactiveerd wordt.

De zelfoptimalisatie leert wanneer het systeem vroeg moet starten om de ruimte op tijd op temperatuur te hebben. De parameter iMinItimeBeforeOperation in de recipe bepaalt de maximaal toegestane voorlooptijd.

De recipe bevat 720 minuten, niet 0 zoals gemeld via het portaal. Als het portaal toch 0 toont, kan de recipe niet correct geladen zijn op de PLC, of levert de kloklogica structureel 0 aan via Globals.iTimebeforeOperKlok4. Te controleren via IXON: live waarde van Globals.iTimebeforeOperKlok4.

3. Ruimtecompensatie

Hoe het werkt

FbCalculatedSupplyTemperature berekent de gewenste aanvoertemperatuur op basis van de buitentemperatuur (stooklijn) én de ruimtetemperatuur. De parameter rRoomCompensationFactor bepaalt hoe sterk de gemeten ruimtetemperatuur bijstuurt. Bij factor 0 is ruimtecompensatie volledig uitgeschakeld.

De aanvoertemperatuur wordt uitsluitend door de stooklijn bepaald, zonder correctie op basis van de werkelijke ruimtetemperatuur. Als de stooklijn te laag uitkomt voor de benodigde warmte, compenseert het systeem niet bij — de ruimte blijft dan onder het setpoint steken.

4. Stookgrens logica

Hoe het werkt

Het systeem gebruikt twee temperaturen tegelijk als stookgrens. Beide moeten boven de ingestelde grens komen voordat het systeem stopt met stoken:

  • Gedempt buitentemperatuur (rTbui_Damp) — kortetermijn gefilterd: 60 meetpunten × tijdslot van 1 uur = buffer van 60 uur
  • Gemiddeld buitentemperatuur (rAveragedOutsideTemperature) — langetermijn voortschrijdend gemiddelde (3-daags)

6. Extra dag prioriteitslogica

Wat bekend is uit de code

De FbScheduler heeft twee inputs die nergens op zijn aangesloten: bParty en bPublicHoliday. De recipe bevat arrays aHoliday[1..5] en aSpecialPeriod[1..n] — alle entries staan op xActivate = FALSE.

🔴 Bug #001: Extra dag overschrijft volledig weekprogramma Gepland

De exacte oorzaak in de code is nog niet gevonden — de WagoAppBuildingHVAC bibliotheek is gesloten (geen broncode inzichtelijk). Het gedrag is in de praktijk bevestigd. Vereist verder onderzoek via IXON-observatie op een dag met een actieve extra dag entry, of via WAGO documentatie over FbScheduler prioriteitslogica.

7. Email / alarm verzending

Oorzaak (issue #005)

De huidige implementatie gebruikt waarschijnlijk de oude WAGO ESMTP_CLIENT library. Moderne mailservers vereisen STARTTLS of TLS — dit ondersteunt de oude library niet. Daarnaast kan de PLC opstarten vóórdat de Ethernet-interface gereed is, wat een TIMEOUT geeft.

Oplossingsrichting

StapActie
1Migreren naar WagoAppMail bibliotheek
2Instelling: smtp.gmail.com, port 587, authenticatie eLOGIN, encryptie eSTARTTLS
3Netwerk gereedheid checken met SOCK_DEVICE_UP vóór verzending
4Gmail App Password gebruiken (geen gewoon wachtwoord meer)
5Uitvoering door Van Niel — vereist codeaanpassing
